# [需求名称] 快速分析（Quick Mode）

> 🚨 **Rana 生成铁律（模型必须严格遵守，违者视为生成失败）**：
> 1. **格式锁**：必须逐字输出下方 4 张 Markdown 表格，❌ 禁止自创列表/分段/更改表头或左侧维度列；
> 2. **UX 视角锚定**：所有分析必须回归“对用户意味着什么”，聚焦场景还原、认知成本、体验一致性、需求真伪，❌ 禁止陷入纯技术实现或业务排期争论；
> 3. <!-- 已注释：Inference-First 原则暂不启用。严禁编造或盲目估算数据/逻辑。 -->
>    **零编造原则**：PRD 未明确的信息或数据，直接标注 `⚠️ 数据缺失` 或 `⚠️ 逻辑未明确`，❌ 禁止自行估算/脑补；所有缺失项必须在 Step 3 提问清单中要求补充；
> 4. **批判强制**：供给维度的“批判性判断”必须包含 `✅ 匹配度 / ⚠️ 逻辑遗漏 / 💡 替代方案` 三要素，❌ 禁止只列功能清单；
> 5. **一句话原则**：每个单元格 ≤2 行，信息密度优先，❌ 禁止背景铺垫/客套话/展开论述；
> 6. **自检放行**：输出前逐条核对上述 5 条，任一不满足 → 立即重写。

## 1. 问题（Who & What Problem）

| 维度 | 内容 |
|------|------|
| 目标用户 | [角色+关键特征+量级]（必须具体，不可写“所有用户”） |
| 具体问题 | [用户视角的障碍，不写产品原因]（例：找不到/易误触/等待焦虑/认知混乱） |
| 发生场景 | [何时何地做什么时遇到，含时机与环境]（例：弱网/单手/排队/跨端切换时） |
| 场景特征 | [频次、操作情境、情绪状态]（例：高频、强时效、容错率低、焦虑感强） |
| 严重程度与影响 | [严重性+影响范围，有数据附数据，无数据标 ⚠️ 数据缺失] |

## 2. 目标（Goal）

| 维度 | 内容 |
|------|------|
| 业务目标 | [关联业务战略的核心目标，不可停留在“优化体验”] |
| 用户目标 | [与第一章“具体问题”直接对应的用户期望] |
| 可量化性 | [指标+目标值，或 ⚠️ 缺乏量化指标（需在提问清单中要求补充）] |
| 冲突与平衡 | [业务vs用户/技术vs体验的冲突点+取舍建议，或联动关系] |

## 3. 供给（Solution）

| 维度 | 内容 |
|------|------|
| 供给内容 | [核心功能/内容/服务，不列边缘需求] |
| 痛点匹配度 | [是否直接解决第一章问题？差距在哪？是否增加认知/操作成本？] |
| 目标达成度 | [能否推动第二章目标？给出用户行为链路上的理由链条] |
| 批判性判断 | ✅ 匹配度：[直击/部分/偏离]；⚠️ 逻辑遗漏：[异常流/边缘场景/迁移成本/体验断裂]；💡 替代方案：[更优路径+优劣对比]（必须完整输出三要素） |

## 4. 指标（Metrics）

| 维度 | 内容 |
|------|------|
| 衡量方式 | [如何衡量，含埋点位置/统计口径/数据采集路径] |
| 具体指标 | [指标名称+明确定义公式]（例：一次成功率=首次扫码成功UV/展示码UV） |
| 基准值 → 目标值 | [当前数据 → 期望数据，缺失必须标 ⚠️ 数据缺失] |

> 🤖 **Rana 填写要求**：严格对应前文四维度，不新增信息；每项 ≤1 句话；聚焦交互设计可落地的抓手与风险。

- ✅ **需求成立判定**：[一句话判定是否成立/能否解决核心问题]

- ⚠️ **关键体验风险**：[最可能引发客诉或流失的体验断点]
