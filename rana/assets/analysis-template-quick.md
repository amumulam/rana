# [需求名称] 快速分析（Quick Mode）

> 🤖 **Rana AI 总体填写原则（Quick Mode）**：
> 1. 本文档定位：快速了解需求、辩证理解、挖掘需求，同时暴露逻辑漏洞、提供评审提问弹药；每个维度只抓核心结论，不展开细节；
> 2. 一句话原则：每个单元格尽量一句话说清，信息密度 > 描述长度；
> 3. Inference-First：PRD 未明确的信息必须推断并标注 `[AI推断]`，不轻易留空；
> 4. 数据缺失必须暴露：无基线数据 → 标 `⚠️ 数据缺失`，绝不编造；
> 5. 批判精神：供给维度必须含专业判断，不能只列功能清单。

## 1. 问题（Who & What Problem）

| 维度 | 内容 |
|------|------|
| 目标用户 | [谁？角色+关键特征+量级]（例：25-35岁高频支付用户，约50万/日） |
| 具体问题 | [用户视角的障碍，不写产品原因]（例：通勤单手操作时，需要3步才能找到常用银行卡，极易误触） |
| 发生场景 | [何时何地做什么时遇到，含时机]（例：早晚高峰在地铁闸机前准备刷码时） |
| 场景特征 | [频次、环境、情境]（例：高频、时间紧迫、单手操作、网络可能不稳定） |
| 严重程度与影响 | [严重性+影响范围，有数据附数据，无数据标 ⚠️ 数据缺失]（例：日均发生800+次，占客诉65%。[AI推断] 导致流失率提升约2%） |

## 2. 目标（Goal）

| 维度 | 内容 |
|------|------|
| 业务目标 | [关联业务战略的核心目标]（例：支撑年度高频用户留存率提升） |
| 用户目标 | [与问题直接对应的用户期望]（例：1秒内快速调出常用银行卡） |
| 可量化性 | [指标+目标值，或 ⚠️ 缺乏量化指标]（例：⚠️ 缺乏量化指标，建议补充"找卡耗时缩短至1秒内"） |
| 冲突与平衡 | [冲突点+取舍建议，或联动关系]（例：业务想推新卡 vs 用户想用老卡。建议：首屏保留老卡，新卡放次屏） |

## 3. 供给（Solution）

| 维度 | 内容 |
|------|------|
| 供给内容 | [核心功能/内容/服务，不列边缘]（例：新增银行卡置顶功能，支持拖拽排序） |
| 痛点匹配度 | [是否直接解决第一章问题？差距在哪？]（例：部分解决。需手动设置置顶，未解决"自动识别常用卡"的诉求） |
| 目标达成度 | [能否推动第二章目标？理由链条]（例：能。置顶后找卡路径从3步缩短至1步，提升留存） |
| 批判性判断 | [更好策略？逻辑遗漏？替代方案优劣？至少回答2项]（例：逻辑遗漏：若用户绑定了10张卡，拖拽排序在小屏幕上极难操作。替代方案：系统基于频次自动置顶推荐） |

## 4. 指标（Metrics）

| 维度 | 内容 |
|------|------|
| 衡量方式 | [如何衡量，含统计口径]（例：埋点统计支付完成率及找卡页面停留时长） |
| 具体指标 | [指标名称+定义]（例：支付转化率 = 完成支付用户数 / 进入收银台用户数） |
| 基准值 → 目标值 | [当前数据 → 期望数据，缺失标 ⚠️]（例：⚠️ 数据缺失 [AI推断] 当前约85% → 期望达到90%） |
