# 需求分析检查清单

## 优先级含义

| 优先级 | 含义 |
|--------|------|
| (P0) | 缺失则无法判断功能方向或用户是谁，设计无法开始 |
| (P1) | 缺失影响设计完整性，进入详细设计前必须确认 |
| (P2) | 缺失不阻断设计，开发前确认即可 |

**Stage 4 执行规则**：所有 P0 项缺口未补完则质量状态为 FAIL；P0 全部补完后，完整性 < 80% 可酌情推进并标注已知风险。

---

在需求分析 Stage 2 中逐项执行。每项标注：
- `✓` 已覆盖（来自 PRD/截图/Figma/确认）
- `✗` 缺失（需在 Stage 3 澄清）
- `⚠` 部分（有但不完整，标注缺失部分）

---

## A. 基本信息

- [ ] A1 需求名称是否明确 (P1)
- [ ] A2 版本/迭代号是否标注 (P2)
- [ ] A3 需求来源是否记录（谁提的、什么时候提的） (P1)
- [ ] A4 业务背景是否说明（为什么要做这个需求） (P0)

## B. 用户与场景

- [ ] B1 目标用户是否定义（谁会用这个功能） (P0)
- [ ] B2 用户角色是否划分（是否有多种角色，权限是否不同） (P1)
- [ ] B3 使用场景是否枚举（用户在什么情况下会用） (P0)
- [ ] B4 场景触发条件是否说明（什么操作/事件触发） (P1)

## C. 流程与状态

- [ ] C1 主流程是否定义（正常情况下的完整操作路径） (P0)
- [ ] C2 分支流程是否枚举（选择不同选项时的不同路径） (P1)
- [ ] C3 异常流程是否覆盖（失败/错误/中断时怎么处理） (P1)
- [ ] C4 状态变化是否定义（对象/页面/按钮的各种状态及转换） (P1)

## D. 边界与约束

- [ ] D1 输入边界是否定义（长度限制、格式要求、必填规则） (P1)
- [ ] D2 数量边界是否定义（列表最大条数、上传文件数量等） (P1)
- [ ] D3 时间边界是否定义（截止时间、有效期、超时处理） (P1)
- [ ] D4 权限边界是否定义（哪些角色可以做哪些操作） (P0)
- [ ] D5 数据边界是否定义（数据范围、数据精度） (P1)
- [ ] D6 技术约束是否标注（平台限制、性能要求、兼容性） (P2)
- [ ] D7 业务规则是否明确（业务逻辑、计算规则、判断条件） (P1)

## E. 异常状态

- [ ] E1 错误状态处理是否定义（操作失败时的提示和处理） (P0)
- [ ] E2 空状态处理是否定义（列表为空/无数据时的显示） (P0)
- [ ] E3 加载状态是否考虑（数据加载中的过渡显示） (P1)
- [ ] E4 网络异常是否处理（断网/超时/请求失败的处理） (P1)
- [ ] E5 权限不足是否处理（无权限时的提示和降级处理） (P1)

## F. 数据与依赖

- [ ] F1 数据实体是否识别（涉及哪些数据对象，如用户/订单/商品） (P1)
- [ ] F2 数据字段是否定义（关键字段名称和类型） (P2)
- [ ] F3 依赖关系是否说明（依赖哪些前置功能或流程） (P1)
- [ ] F4 第三方服务依赖是否标注（外部接口、第三方 SDK） (P2)

## G. 信息架构

- [ ] G1 页面结构是否推断（需要几个页面，页面层级关系） (P1)
- [ ] G2 导航层级是否识别（从哪里进入，返回路径） (P1)
- [ ] G3 内容组织是否说明（页面内的信息优先级和布局逻辑） (P2)

## H. 可追溯性

- [ ] H1 每项分析是否标注来源（[PRD第X节] / [截图X] / [推断] / [缺失]） (P0)
- [ ] H2 来源类型是否区分（原始文档 vs 推断 vs 设计师补充） (P1)
