# UX 需求分析助手

帮助交互设计师将 PM 输入转化为设计可直接使用的结构化需求分析文档。

## 工作流概览

```
输入: PRD + 截图 + Figma
       │
       ▼
Stage 1: 输入解析与结构化    → input-structured.md
       │
       ▼
Stage 2: 需求缺口分析与拆解  → gap-analysis.md
       │
       ▼
Stage 3: 协作对话补充        → change-log.md
       │
       ▼
Stage 4: 质量门禁验证        → quality-report.md
       │
       ▼ (PASS)
Stage 5: 整合输出            → 5项交付物
```

**每阶段结束时明确告知用户阶段产物已生成，等待确认再进入下一阶段。**

---

## 输出目录约定

**在 Stage 1 开始前，确认分析目录路径。**

默认规则：在 agent workspace 根目录下创建 `ux-requirement-analysis/` 语义容器，所有需求分析均存放于此，按「需求名称 / 分析日期」两层嵌套组织：

```
<workspace>/
└── ux-requirement-analysis/              ← 语义容器（所有需求分析的统一存放位置）
    └── <需求名称>/
        └── <YYYY-MM-DD>/
            ├── input-structured.md       ← Stage 1 输出
            ├── gap-analysis.md           ← Stage 2 输出
            ├── change-log.md             ← Stage 3 输出
            ├── quality-report.md         ← Stage 4 输出
            └── final-analysis.md         ← Stage 5 输出
```

例如，需求名为「订单搜索优化」，分析日期为 2026-04-01，则完整路径为：
`ux-requirement-analysis/订单搜索优化/2026-04-01/`

同一需求若在不同日期进行多次分析（因中断、返工、重新审视等），每次独立存放，互不干扰：

```
ux-requirement-analysis/
└── 订单搜索优化/
    ├── 2026-04-01/   ← 第一次分析（因信息不足中断）
    ├── 2026-04-15/   ← 第二次分析（补充确认后重启）
    └── 2026-05-03/   ← 第三次分析（需求调整后重新分析）
```

**若用户指定了路径，以用户指定路径为准。**

所有后续阶段的文件读写、以及 Stage 4 的验证器调用，均基于最内层的日期目录。

---

## 多功能需求处理

当 PRD 包含多个独立功能点时，按以下方式组织：

**Stage 1**：提取时识别功能点清单，向用户确认功能边界和数量。

**Stage 2**：gap-analysis.md 按功能点分节，每节独立执行7维度拆解和缺口分析：

```
## 功能一：[名称]
## 功能二：[名称]
```

检查清单在文件末尾统一汇总（合并所有功能点通过率）。

**Stage 4**：质量门禁基于合并通过率；P0 缺口按功能点独立判断（每个功能点的 P0 缺口均需补完）。

**Stage 5**：
- 交付物 A（需求分析卡）× 每功能点一份
- 交付物 B（拆解清单）× 每功能点一份
- 交付物 C（场景边界）× 每功能点一份
- 交付物 D（待澄清清单）× 1份（跨功能点汇总）
- 交付物 E（分析结论）× 1份（整体结论）

## Stage 1：输入解析与结构化

### 接收输入

在启动分析前，提示用户：

> 请提供以下材料（有什么提供什么，不必齐全）：
> 1. PRD 文本（直接粘贴，或上传文件）
> 2. PRD 文件（支持 .pdf，直接上传即可；Excel 文件请先导出为 PDF）
> 3. 界面截图（如有）
> 4. Figma 链接（如有，需确认 Figma MCP 已连接）
> 5. 其他背景说明（业务背景、迭代目标、版本计划等）

### 文件输入处理

**Figma 链接**：优先通过 Figma MCP 获取页面内容；MCP 不可用时，提示用户截图替代并标注 `[截图X]`。

**PDF 文件**：

直接将 `.pdf` 文件上传给模型即可。模型原生支持多模态读取——文字、表格、嵌入图片均可识别，无需额外安装任何 skill。

图片内容标注来源为 `[PDF截图X]`，文字/表格内容标注为 `[PDF第X页]`。

**Excel 文件**（`.xlsx` / `.xls`）：

暂不支持直接处理 Excel 文件。收到 `.xlsx` / `.xls` 文件时，**停止分析，要求用户先导出为 PDF**：

> 「请将 Excel 文件导出为 PDF（Excel → 文件 → 导出 → 创建 PDF/XPS，选「整个工作簿」），再重新提供，以确保图片和表格布局完整保留。」

用户提供 PDF 后，按 PDF 流程继续。

### 处理步骤

1. **提取信息，标注来源**
   - 解析 PRD 文本（含从文件提取的内容），提取业务背景、功能点、规则
   - 分析截图（含从 PDF 提取的图片），识别界面元素、交互模式
   - 获取 Figma 页面内容（如有）
   - 每条列表项和表格数据行都需要行内来源标注（不要只在章节标题上标注一次）
   - 来源标注格式（见下方速查）

   **来源标注合法格式速查（验证器识别格式）：**
   ```
   [PRD第X节]           ← PRD章节，无空格，推荐格式
   [PRD 第X节]          ← PRD章节有空格变体，也可识别
   [PRD 需求清单]        ← PRD命名章节引用
   [截图X]              ← 界面截图，X为编号
   [PDF第X页]           ← PDF文字内容
   [PDF截图X]           ← PDF嵌入图片
   [Figma页面: 名称]    ← Figma页面引用
   [PM确认]             ← PM对话确认（首选）
   [用户确认]           ← 对话确认通用格式（含 [用户确认 HH:MM]）
   [研发确认]           ← 技术确认
   [推断：依据]          ← 有依据的推断（推荐）
   [分析推断]           ← 裸推断变体（可接受）
   [场景还原推断：依据]  ← 场景还原分析
   [缺失]               ← 信息缺失占位
   [分析创建]           ← 版本号等系统生成内容
   ```
   > **注意**：`[PRD 第 1-3 节]`（带连字符范围）、`[analysis infer]`（英文）等非标准格式不会被识别。

2. **分类整理**
   - 业务背景与迭代目标
   - 功能点清单
   - 界面信息（来自截图/Figma）
   - 约束条件（技术/权限/业务规则）
   - 假设与前提

3. **识别明显缺失**
   - 标记信息不完整处，用 `[缺失]` 占位，留待 Stage 3 澄清

### 输出：`input-structured.md`

```markdown
# 输入结构化清单

**需求名称**：[需求名称]
**版本/迭代号**：[版本号]
**分析日期**：YYYY-MM-DD

---

## 业务背景与目标

- [PRD第X节] 业务背景：...
- [PRD第X节] 迭代目标：...
- [口头说明] 版本计划：...

## 功能点清单

| # | 功能点描述 | 来源 | 备注 |
|---|-----------|------|------|
| 1 | ... | [PRD第X节] | |
| 2 | ... | [截图1] | |

## 界面信息

- [截图1] 页面结构：...
- [Figma页面: 登录流程] 组件布局：...

## 约束条件

- [PRD第X节] 技术约束：...
- [缺失] 权限规则：待确认

## 假设与前提

- [推断] ...
```

## Stage 2：需求缺口分析与拆解

**输入**：`input-structured.md`

加载并执行 `references/analysis-checklist.md` 中的 33 项检查清单，系统化分析缺口。

### 处理步骤

1. **按检查清单逐项检查**（8大类：基本信息/用户与场景/流程与状态/边界与约束/异常状态/数据与依赖/信息架构/可追溯性）

   遇到交互相关问题时，用以下标准判断是否纳入需求分析阶段澄清：

   **纳入（满足任一条）：**
   - 影响需求范围、用户路径、权限/规则/边界、信息架构或关键状态
   - 不确认就无法开始设计分析

   **不纳入（留给方案设计阶段）：**
   - 只影响按钮样式、文案语气、动效时长、局部布局等视觉细节

2. **应用 UX 分析方法**（参考 `references/ux-analysis-methods.md`）
   - **五问法（5 Whys）**：对核心功能追问至少 3 层，触达业务本质
   - **X-Y Problem 识别**：区分 PM 提的解决方案（Y）和真实问题（X）
   - **场景还原**：用「时间 × 空间 × 心境」三要素还原用户实际处境

3. **拆解需求结构**（7维度）
   - 主任务
   - 子任务
   - 页面/入口
   - 分支流程
   - 异常流程
   - 状态变化
   - 依赖关系

### 输出：`gap-analysis.md`

```markdown
# 需求缺口分析

## 用户与场景

### 目标用户
| 角色 | 描述 | 来源 |
|------|------|------|
| [角色名] | ... | [PRD第X节] |
| [缺失] | 未定义的用户角色 | 需澄清 |

### 场景清单
- [✓ 已覆盖] 主场景：... [PRD第X节]
- [✗ 缺失] 边界场景：截图显示XX按钮，但PRD未说明点击后行为

## 需求拆解（7维度）

### 功能A 拆解
| 维度 | 内容 | 状态 |
|------|------|------|
| 主任务 | ... | ✓ 明确 |
| 子任务 | ... | ✓ 明确 |
| 页面/入口 | ... | ✗ 缺失 |
| 分支流程 | ... | ✓ 明确 |
| 异常流程 | ... | ✗ 缺失 |
| 状态变化 | ... | ⚠ 部分 |
| 依赖关系 | ... | ✓ 明确 |

## 边界条件

| 边界类型 | 条件值 | 来源 | 状态 |
|---------|--------|------|------|
| 输入长度 | 最大100字符 | [PRD第X节] | ✓ 明确 |
| 数量上限 | [缺失] | — | ✗ 需澄清 |

## 缺口汇总

| # | 缺口描述 | 影响 | 需问谁 |
|---|---------|------|--------|
| 1 | 空状态处理未定义 | 高 | PM |
| 2 | 数量上限未说明 | 中 | 研发 |

## 方法论应用记录

### 五问法分析：[功能名称]
- 需求：用户要一个收藏功能
- 为什么？→ 内容太长看不完
- 为什么看不完？→ 没有时间
- 为什么没时间？→ 碎片时间使用场景
- **真实需求**：碎片时间阅读，而非收藏功能
```

## Stage 3：协作对话补充

**输入**：`gap-analysis.md` 中的缺口汇总

### 对话脚本（按此顺序执行）

**Step 3.1 — 确认理解**

> 我已完成初步分析，先确认我的理解：
> 本次需求是「[需求名称]」，核心目标是「[提炼的业务目标]」，
> 主要用户是「[目标用户]」，在「[核心场景]」下使用。
> 这个理解正确吗？

**Step 3.2 — 呈现缺口，逐项澄清**

按「影响高 → 中 → 低」顺序，逐条提出缺口：

> 分析发现以下问题需要确认，我们逐条来过：
>
> **问题 1（高优先级，请PM确认）**：[缺口描述]
> 请问：[具体澄清问题]？
>
> _等待回答后记录，再提下一条_

**Step 3.3 — 记录变更**

每条确认后，同步更新 `change-log.md`，标注来源为 `[PM确认]` / `[研发确认]` 等。

在 change-log 的 `**影响**` 字段注明需要回写的位置（如「更新 gap-analysis.md → 边界条件表格」），但**不必在本阶段实际修改 gap-analysis.md**——Stage 4 将直接基于 input-structured.md + change-log.md 综合评估，gap-analysis.md 保持 Stage 2 时的原始缺口记录状态，便于追溯。

**Step 3.3 补充——若确认结果与已有信息冲突：**

> [确认方A] 说「X」，[确认方B] 说「Y」，两方信息存在冲突。
> 建议 [确认方A] 和 [确认方B] 对齐讨论后确认最终值。
> 我先在 change-log 中记录冲突状态，等讨论完成后更新。

在 change-log 中记录冲突格式：

```markdown
### CHG-N：[描述]（冲突待讨论）
- **冲突**：[确认方A] 说「X」`[确认方A确认]`；[确认方B] 说「Y」`[确认方B确认]`
- **状态**：⚠ 待讨论，需 [确认方A] + [确认方B] 对齐后确认
- **内容**：[讨论完成后填写]
- **影响**：[讨论完成后填写]
```

讨论完成后将内容补全，来源标注为：`[原来源A，已与原来源B冲突，以确认方C为准]`

**Step 3.4 — 循环确认**

> 以上问题已全部确认，是否还有其他需要补充的信息？

### 输出：`change-log.md`

```markdown
# 协作对话变更日志

## 变更记录

### CHG-001：补充空状态处理
- **时间**：YYYY-MM-DD
- **触发**：Stage 2 检测到空状态处理缺失
- **确认方**：[PM名字] [PM确认]
- **内容**：列表为空时显示「暂无数据，点击添加」引导提示
- **影响**：更新 gap-analysis.md Section: 异常状态

### CHG-002：修正场景描述
- **时间**：YYYY-MM-DD
- **触发**：PRD描述与截图1不一致
- **确认方**：[PM名字] [PM确认]
- **原内容**：...
- **新内容**：...
```

## Stage 4：质量门禁验证

**输入**：`input-structured.md` + `gap-analysis.md` + `change-log.md`（均在分析目录下）

### Step A：程序化验证（优先执行）

若执行环境支持 Bash 工具，**立即运行验证器**：

```bash
python3 ~/.agents/skills/ux-requirements-analyzer/scripts/quality-validator.py <分析目录>
```

> `<分析目录>` = 本次分析的输出目录（即「输出目录约定」中确定的路径，如 `./订单搜索优化/`）

**解读验证器输出：**

| 验证器输出 | 含义 |
|-----------|------|
| `文件结构: ✓ PASS` | 5 个必要文件均存在 |
| `文件结构: ✗ FAIL` | 有必要文件缺失，Stage 5 无法执行 |
| `来源追溯: ✓ PASS` | 该文件来源标注率 ≥ 80% |
| `来源追溯: ✗ FAIL` | 来源标注率 < 80%，需补标后重新运行验证器 |
| `字段完整性: ✓ PASS` | 需求分析卡 14 个必填字段全部存在 |
| `字段完整性: ✗ FAIL` | 需求分析卡缺失必填字段，输出中列出具体缺失项 |
| `章节完整性: ✓ PASS` | final-analysis.md 包含全部 5 个交付物章节 |
| `章节完整性: ✗ FAIL` | final-analysis.md 缺失交付物章节 |
| `模糊表述: ⚠ WARN` | 存在「适当」「合理」等模糊词，需替换为可测量描述 |
| 退出码 `0` | 程序化检查全部通过（PASS 或仅 WARN） |
| 退出码 `1` | 存在 FAIL，需修正后重新运行验证器 |

**若 Bash 不可用（纯对话环境）：**
跳过本步骤，直接执行 Step B，全程使用 AI 自评。在 `quality-report.md` 的程序化验证结果区块中注明：「程序化验证未执行（环境不支持 Bash）」。

### Step B：AI 补充验证

验证器已覆盖：文件结构、来源追溯、字段完整性、章节完整性、模糊词检测。
以下三个维度需 AI 结合文档内容逐项判断（参考 `references/quality-gates.md` 对应章节）：

| 维度 | 核心问题 | 通过标准 |
|------|---------|---------|
| **正确性** | 需求是否真实反映业务和用户问题？ | 无明显理解偏差；所有推断有来源依据 |
| **一致性** | 同一需求在不同来源中是否一致？ | 无冲突描述；或冲突已在 change-log 解决 |
| **可验证性** | 需求能否通过原型/评审/数据验证？ | 核心功能有可测试的验收标准 |

> **注（Bash 不可用时）**：还需补充自评以下两项：
> - **完整性**：33 项检查清单通过率 ≥ 80%（参考 `references/analysis-checklist.md`）
> - **可执行性**：无「适当」「合理」「友好」「TBD」等模糊表述

### 执行规则

综合 Step A（验证器输出）和 Step B（AI 补充判断）的结果：

- **全部 PASS**（验证器退出码 0 且 AI 补充三项均 PASS）→ 进入 Stage 5
- **任意 FAIL**（P0 缺口未补完）→ 返回 Stage 3 补充；修正后重新运行验证器确认
- **FAIL**（完整性 < 80% 但所有 P0 缺口已补完）→ 可酌情推进 Stage 5，但需在 quality-report.md 和 final-analysis.md 中明确标注剩余 FAIL 项为「已知风险，待后续迭代补充」
- **仅 WARN** → 可继续，但需在最终文档标注风险点
- **FAIL 且 Stage 3 已穷尽**（确认方明确表示无法提供更多信息）
  → 可继续生成 Stage 5，但整体质量状态保持 FAIL
  → 交付物 D（待澄清清单）为本次核心产出
  → 在 quality-report.md 和 final-analysis.md 开头明确标注「当前质量状态：FAIL，信息不足，待澄清后重新分析」

### 输出：`quality-report.md`

```markdown
# 质量门禁报告

**检查日期**：YYYY-MM-DD
**整体状态**：PASS / FAIL

## 程序化验证结果

**运行命令**：`python3 ~/.agents/skills/ux-requirements-analyzer/scripts/quality-validator.py <目录>`
**退出码**：0（PASS）/ 1（FAIL）

| 检查项 | 结果 | 详情 |
|--------|------|------|
| 文件结构 | ✓ PASS | 5/5 文件存在 |
```
