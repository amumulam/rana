# 需求分析交付物模板

## 1. 阶段定义

**需求分析**：交互设计师在接收 PM 的需求清单、PRD、口头说明或草图后，对需求进行理解、拆解、澄清、归因与结构化，明确本次迭代要解决的问题、目标用户、使用场景、业务边界、约束条件和待确认项，为后续设计分析提供可执行输入。 [woshipm](https://www.woshipm.com/ucd/1340798.html)

***

## 2. 输入材料

需求分析阶段至少应接收以下输入：

- PM 输出的需求清单或 PRD。 [lanlanwork](http://www.lanlanwork.com/blog/post-3798.html)
- 业务背景说明、迭代目标、版本计划。 [lanlanwork](http://www.lanlanwork.com/blog/post-3798.html)
- 用户反馈、数据报表、工单、客服反馈、竞品参考。 [woshipm](https://www.woshipm.com/rp/391784.html)
- 技术约束、权限规则、合规要求、埋点现状。 [blog.csdn](https://blog.csdn.net/m0_54786252/article/details/131389916)

***

## 3. 分析方法

需求分析阶段建议按以下顺序执行：

1. **理解需求原文**：确认 PM 想解决什么问题，以及需求描述是否完整。 [woshipm](https://www.woshipm.com/ucd/1340798.html)
2. **拆解需求结构**：拆成目标、用户、场景、流程、状态、规则、异常、依赖。 [m.zcool.com](https://m.zcool.com.cn/article/ZMTI3NzkyNA==.html)
3. **澄清歧义和缺口**：把不清楚、冲突、遗漏、不可执行的部分提出来。 [testwo](https://www.testwo.com/blog/5852)
4. **归纳核心问题**：把零散功能描述提炼成设计可处理的问题定义。 [woshipm](https://www.woshipm.com/rp/391784.html)
5. **形成结构化结论**：输出后续设计分析可直接使用的设计输入。 [blog.csdn](https://blog.csdn.net/m0_54786252/article/details/131389916)

| 约束条件 |  |
| 优先级 |  |
| 待澄清项 |  |
| 风险点 |  |
| 需求结论 |  |

#### B. 需求拆解清单
把 PRD 或需求清单拆成交互可处理的颗粒度：

- 主任务。
- 子任务。
- 页面/入口。
- 分支流程。
- 异常流程。
- 状态变化。
- 依赖关系。 [m.zcool.com](https://m.zcool.com.cn/article/ZMTI3NzkyNA==.html)

#### C. 场景与边界说明
说明“在哪些情况下用户会用、在哪些情况下不能用、有哪些限制”。

- 典型使用场景。
- 边界场景。
- 不支持场景。
- 业务/技术/权限限制。 [testwo](https://www.testwo.com/blog/5852)

#### D. 待澄清问题清单
把未确认内容显式列出，确保需求不带病进入设计。

- 需要 PM 确认的问题。
- 需要研发确认的问题。
- 需要测试确认的问题。
- 需要业务确认的问题。 [lanlanwork](http://www.lanlanwork.com/blog/post-3798.html)

#### E. 需求分析结论
用 1 到 3 句话总结本次需求分析结果，建议写成：

- 谁。
- 在什么场景。
- 遇到什么问题。
- 本次要优先解决什么。 [woshipm](https://www.woshipm.com/ucd/1340798.html)

## 5. 可选交付物

如果需求较复杂，可补充以下内容：

- 用户路径图。
- 异常流程图。
- 需求关系图。
- 风险清单。
- 数据/埋点建议。
- 需求优先级说明。 [lrd](https://lrd.im/project/analytics-guide)

***

## 6. 完成标准

需求分析只有满足以下条件，才算完成：

- 需求背景已明确。
- 核心问题已定义。
- 用户和场景已明确。
- 主流程、分支流程、异常流程已拆解。
- 关键边界与约束已标注。
- 待澄清项已列出并推动确认。
- 需求结论可直接进入设计分析。 [blog.csdn](https://blog.csdn.net/m0_54786252/article/details/131389916)

***

## 7. 审核标准

建议内部评审时重点看以下几个维度：

### 正确性
需求是否真实反映业务和用户问题，是否理解偏差。 [woshipm](https://www.woshipm.com/rp/391784.html)

### 完整性
主流程、异常、边界、依赖、约束是否都被覆盖。 [testwo](https://www.testwo.com/blog/5852)

### 可执行性
是否能直接支撑后续设计分析和方案设计。 [lanlanwork](http://www.lanlanwork.com/blog/post-3798.html)

### 一致性
同一需求在不同文档、口径、评审中是否一致。 [blog.csdn](https://blog.csdn.net/m0_54786252/article/details/131389916)

### 可验证性
需求是否能通过原型、评审、测试或数据指标验证。 [woshipm](https://www.woshipm.com/rp/391784.html)

## 8. 推荐命名方式

如果放到团队文档里，建议叫：

- **XX 需求分析**
- **XX 需求交互分析**
- **XX 需求分析与设计输入**
- **XX 迭代需求澄清**

如果你们想强调交互角色，推荐最后一个 **“需求分析与设计输入”**，最不容易和“设计分析”混淆。 [woshipm](https://www.woshipm.com/ucd/1340798.html)
