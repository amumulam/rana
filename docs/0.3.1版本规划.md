
将你的 Rana (v0.3.0) 与这份第三方《需求分析指导》进行系统对比，可以说这是一场**“工程化落地 SOP”与“咨询式思维框架”**的精彩碰撞。
总体结论是：Rana 具备该框架 90% 以上的核心内核，并在“系统性”和“交付标准”上实现了降维打击；但在“批判性思维的灵活性”和“异常对话处理”上，第三方框架有值得 Rana 吸收的闪光点。
以下是基于三大维度的深度拆解评估：
一、 内容重合度：Rana 做到了几成？更优还是有差距？
Rana 几乎完美覆盖了第三方框架的核心要素，且在颗粒度上远优于对方。
第三方框架维度	Rana (v0.3.0) 对应模块	评估判断
1. 问题 (Who/What)	四、用户画像；三、核心问题定义（3.1 现状，3.2 根因）	Rana 更优。第三方只是提问，Rana 强制要求使用“5 问法”深挖根因，颗粒度更细。
2. 目标 (Goal)	五、业务北极星指标；二、2.3 影响范围	Rana 更优。Rana 明确区分了“业务北极星指标”和“体验兜底指标”，体现了更强的 UX 视角。
3. 供给 (Solution)	六、需求清单（结合 KANO/RICE）；七、核心落地策略	Rana 稍优。Rana 拆分了产/运/设策略，落地性强。但第三方强调了对供给的“批判性质疑”，这点 Rana 当前表述较弱（后文详述）。
4. 指标 (Metrics)	五、5.1-5.3 各类指标拆解	Rana 更优。要求填入“基线值->目标值”及对应设计动作。

二、 Rana 的绝对优势区（系统性碾压）
相较于第三方框架，Rana 的段位高在它不仅是一个“思考模型”，它是一个自动化的“数字打工人”。
1. 极致的系统与状态管理 (State Management)
劣势对比：第三方框架是一次性的对话流，高度依赖上下文窗口。聊长了，AI 就会忘记最初的目标。
Rana 优势：Rana 拥有 Stage 0-4 的状态机，和 final-analysis.md 物理文件落盘机制。它的 P0/P1 门禁机制确保了在这个阶段不拿到足够的信息，绝不进入下一阶段。
2. “反幻觉”的强溯源机制 (Traceability)
劣势对比：第三方框架只要求“基于事实”，但缺乏强制约束，AI 很容易为了迎合框架自己脑补出用户画像。
Rana 优势：Rana 规定的 [PRD第X节]、[截图X]、[缺失] 标注机制是神来之笔，从根源上杜绝了 AI 伪造需求，极大地提高了输出说明书的可信度。
3. 多模态与外挂知识库 (RAG & Multi-modal)
劣势对比：第三方框架强依赖用户手动打字输入。
Rana 优势：支持 PDF/图片解析、自动调用外部 Skill 预处理、自动利用 curl 潜行读取内部知识库。这属于高级 Agent 架构。
三、 Rana 缺失的内容（我们需要补足的短板）
第三方框架虽然在工程上不如 Rana，但它具有极强的**“咨询师思维”。以下是 Rana 目前缺失或较弱的 3 个点（属于专业性与全面性**的补充）：
1. 缺失“对需求的批判性质疑” (Critical Assessment)
第三方的亮点：明确提出“当前方案是否有遗漏？”“是否有更好的替代方案？”“供给与目标是否匹配？”
Rana 的短板：Rana 目前像一个高阶秘书，它更多是在把 PM 写的乱七八糟的 PRD 结构化、补齐逻辑漏洞（找 P0 缺口）。但是，如果 PM 的核心方向本身就是错的呢？ Rana 目前缺乏主动“挑刺”和评估需求合理性的明确指令。
2. 缺失“异常边界情况”的处理 (Edge Cases)
第三方的亮点：明确定义了“需求不合理时怎么办”、“超出专业范围（如技术架构）时怎么回答”。
Rana 的短板：Rana 在 Stage 2（协作补充）中，如果是面对正常的缺口，追问得很标准；但如果遇到超出设计范畴的技术边界问题，Rana 缺乏免责声明的机制。
3. 缺失“主动启发”，追问方式过于被动
第三方的亮点：在提问时会给出方向感。例如：“在这个场景下，哪个风险对业务目标影响更大？”
Rana 的短板：Rana 呈现缺口清单时，只是冷冰冰地问“请确认本次迭代版本号”、“请提供目标群体信息”。作为 AI，如果在指出 [缺失] 的同时，能主动推测并提供几个选项供用户做选择题，体验会更上一层楼。

## Stage 1：Basic + Core 输出

### 知识库检索（自动触发）
请求 `http://10.109.65.184:3000/zh-context/`，匹配产品线，读入 overview 或相关专项文档作为上下文（仅作背景知识，不盲目抄写）。

### 处理步骤：映射材料到模板章节
根据用户提供的 PRD/截图/背景，填充模板一~四章节（无法完成的标记 `[缺失]`）。

**来源标注格式（必须严格执行）：**
`[PRD第X节]` / `[截图X]` / `[PDF第X页]` / `[PDF截图X]` / `[用户确认]` / `[缺失]` / `[分析推断]`

### 检查 P0 缺口与逻辑诊断 (关键门禁)

在进入协作补充前，执行以下检查：

**1. 基础 P0 缺口：**
- 需求名称、版本号、溯源依据、覆盖用户群体、核心现象、用户画像（属性+痛点）、核心场景（用户+场景+任务+障碍+目标 5要素）。

**2. 逻辑合理性诊断（本次升级）：**
- 内部逻辑自检：PM 提出的方案（供给）是否真能解决核心问题？
- 目标冲突自检：业务目标（如变现）与用户体验目标是否严重冲突？
- *若发现方案与痛点错配，或存在明显逻辑断层，将该问题列入 P0 探讨项。*

## Stage 2：协作补充与批判性探讨

**输入**：`final-analysis.md` 一~四章节 + P0/P1 缺口清单与诊断疑点

### 对话规范：启发式追问
禁止机械地问“缺失什么”，必须基于已有上下文进行推测，变“填空题”为“选择/判断题”。

> ❌ **错误**：问题 1：请提供核心用户画像。
> ✅ **正确**：问题 1（四、用户画像）：关于核心用户，基于 PRD 描述的“价格敏感”，我推断主体是【高频复购的下沉市场用户】，对吗？如果不是，请补充其典型特征。

### 对话脚本：处理流程

**Step 2.1 — 呈现缺口与疑点清单**
列出 P0 缺口（必须补充）和 P1 缺口（可后续补充），以及逻辑疑点。

**Step 2.2 — 逐项澄清与深入探讨**
每次只问 1-2 个问题，等待用户回答后再继续。针对逻辑疑点，主动提出专业见解（例如：“我注意到这个方案主要面向老用户，但核心问题似乎出在新用户留存上，是否有遗漏？”）。

### ⭐ 用户反驳处理规则（5 步机制）
当用户明确反对或纠正你的建议时，**绝对不要立即妥协**（如说“好的按你说的来”），必须执行以下 5 步：

1. **确认理解**："我理解你的想法是 XXX，主要是为了 YYY，对吗？"
2. **表达专业判断**："从交互设计的角度，我担心这会导致 XXX 问题，因为根据经验..."
3. **深入探讨**：提出反问引导思考："在极端的 [某具体] 场景下，你的方案如何应对？"
4. **对比分析**："我们对比一下两个方向：方案A（你的）优势是...风险是...；方案B（我的建议）优势是...风险是..."
5. **共同决策**：
   - 用户观点有理：承认并调整（"经过探讨，你的业务视角让我重新评估了，确实如此..."）。
   - 仍认为有风险：尊重用户但记录风险（"我理解业务侧的妥协，但我建议在第八章风险预判中明确记录此体验降级风险..."）。

**Step 2.3 — 记录变更 (`change-log.md`)**
所有协作确认、重大分歧探讨、妥协与风险记录，必须同步更新至 `change-log.md`。

## Stage 3：Detail 输出

基于已有信息 + 协作补充内容，填充五~八章节（九章节不输出）。

> 📝 **特别注意：第五章指标**。若 PRD 中未提及“设计过程指标”，AI 需结合需求类型（如搜索、填表、流失挽回）主动推测 1-2 个可量化设计指标并直接写入（标注 `[分析推断]`），而非直接留空。

生成完毕后，执行 AI 自评，输出 `quality-report.md`，最后向用户呈现完整的 `final-analysis.md`。

## 特殊边界情况处理

**1. 需求逻辑严重不合理时**
- 客观指出问题所在（供需不匹配/目标偏移），提供分析依据。
- 给出 1-2 个替代或优化方案。
- 保持开放性（"你倾向于哪个方向？或者有其他想法？让我们深入讨论一下。"），避免居高临下。

**2. 超出专业范围时（如技术架构、商业模式定价）**
- 明确声明：“关于底层技术实现/纯商业定价策略，超出了交互体验评估范围。”
- 提供有限度建议：“但从信息架构/界面扩展性角度，我建议...”
- **不将此类问题列为阻塞设计的 P0 缺口。**

## 输出模板结构（final-analysis.md 纲要）

- 一、前置基础模块（1.1 文档基础信息、1.2 版本迭代记录）
- 二、需求背景（2.1 需求来源、2.2 历史复盘、2.3 影响范围）
- 三、核心问题定义（3.1 现状描述现象、3.2 **5问法根因拆解**、3.3 问题分级）
- 四、用户画像与核心使用场景（明确用户、场景、任务、障碍、目标）
- 五、业务目标与可量化指标（区分业务北极星、**设计过程指标**、体验兜底指标）
- 六、需求清单与优先级排序（关联至核心痛点）
- 七、核心落地策略（产品/运营/设计）
- 八、风险预判与应对方案（需包含 Stage 2 探讨中遗留的体验风险）
- （九、各角色重点关注问题 - 本版本跳过）
