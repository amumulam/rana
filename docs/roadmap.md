| F8 | 第七章方案批判性评估 | P0 | 规划中 | 新增 7.5 小节：有效性评估、替代方案、决策建议 |

#### F7: 四大维度关联校验

**核心任务**:
- 校验对象：所有 P0 级需求（P1/P2 仅提示，不强制）
- 关联要求：每个 P0 需求必须对应：
  - 关联「第三章 核心问题定义」中的具体核心问题
  - 关联「第五章 业务目标」中的具体目标
  - 关联「第五章 指标」中的具体衡量指标
- 校验时机：Stage 3 Detail 输出完成后、Detail P0 缺口检查前
- 缺口标记：未关联/关联错误的 P0 需求标记为 Detail P0 缺口

**验收标准**:
- 所有 P0 需求未关联时 100% 触发校验缺口
- 需求清单表格新增「关联项」列
- 关联错误能被识别并标记

#### F8: 第七章方案批判性评估

**新增位置**: 第七章 7.4 之后，作为 7.5 小节

**强制内容**:
- 当前方案有效性评估（能否解决问题、达成目标）
- 潜在问题与遗漏点（至少 1 个）
- 替代方案对比（表格形式）
- 最终决策建议

**验收标准**:
- 未填写或内容不达标触发 P0 缺口校验
- 替代方案具备可行性
- 与 Stage 2 反驳机制衔接

#### F1: 同需求多轮迭代关联

**问题**: 当前同名需求多次分析独立存放，无法继承上一版有效内容、无法标记变更差异、无法关联历史文档。

**方案**:
- 增量继承：检测同名需求历史目录，提示用户选择「继承上一版」或「全新分析」
- 变更标记：继承模式下沿用内容标注「[继承V1.0]」，新增/修改内容正常标注来源
- 版本迭代记录关联：继承上一版版本迭代记录表，追加本次迭代条目
- 历史文档链接：在 `final-analysis.md` 头部添加「关联历史版本」字段

**目录结构**:
```
ux-requirement-analysis/
└── 订单搜索优化/
    ├── 2026-04-01/   ← V1.0（首次分析）
    ├── 2026-04-15/   ← V1.1（继承V1.0，增量更新）
    └── 2026-05-03/   ← V2.0（需求调整后全新分析）
```

#### F2: 特殊需求类型适配

**问题**: 当前流程围绕「功能迭代类需求」设计，未针对不同需求类型做差异化适配。

**需求类型分类**:

| 类型 | 核心章节权重 | P0 缺口调整 | 流程适配 |
|------|-------------|------------|---------|
| 功能迭代类（默认） | 一~八全量 | 标准 P0 | 标准 4 阶段 |
| BUG 修复类 | 三权重高；四权重低 | 用户画像/场景降为 P1 | Stage 1 仅输出一~三章节，Stage 2 快速通过 |
| 合规整改类 | 二、七权重高 | 合规红线为强制 P0 | 增加「合规清单」检查项，合规确认阻塞 |
| 运营活动类 | 五、四权重高 | 活动目标/指标为 P0 | Stage 3 侧重运营策略，策略模板调整 |
| ToB 大客户定制类 | 六、八权重高 | 客户确认项为 P0 | 增加客户确认环节，需求清单关联客户诉求 |

**技术要点**:
- `config.yaml` 新增字段：`requirement_type: functional | bug_fix | compliance | operation | to_b`
- SKILL.md 各阶段根据类型动态调整 P0 检查清单
- Stage 1 输出模板根据类型选择性输出章节

## 版本规划原则

### 优先级定义

| 级别 | 定义 | 触发条件 |
|------|------|----------|
| P0 | 阻塞发布 | 核心流程缺陷、数据丢失风险 |
| P1 | 本版本必须 | 缺失影响核心用例 |
| P2 | 建议补充 | 提升专业度、用户体验 |
| P3 | 远期规划 | 超出当前版本范围 |

### 版本号规则

- **主版本号 (x.0.0)**: 核心流程重大变更
- **次版本号 (0.x.0)**: 新功能特性
- **修订号 (0.0.x)**: Bug 修复、文档更新
