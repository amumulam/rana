**日期**: 2026-03-31
**版本**: v1.0
**作者**: Claude + User协作

## 1. 项目概述

### 1.1 背景

交互设计师在接收 PM 的 PRD/需求清单后，需要进行需求分析：理解、拆解、澄清、归因与结构化，明确问题、目标用户、使用场景、业务边界、约束条件和待确认项，为后续设计分析提供可执行输入 <sup>[1]</sup>。

目前市场上缺少专为此场景设计的 AI 能力：
- 现有需求分析工具面向 PM/BA/开发者，不是设计师视角
- 缺少多模态输入处理（PRD + 截图 + Figma）
- 输出产物不够结构化，难以直接进入设计分析阶段

### 1.2 目标

设计一个 Skill 插件，帮助交互设计师：
1. 接收多模态输入（PRD 文本、截图、Figma 链接）
2. 自动化执行需求分析流程 <sup>[2]</sup>
3. 输出结构化、可追溯、即用的需求分析文档
4. 通过质量门禁保障输出质量

### 1.3 核心价值主张

| 维度 | 目标 |
|---|---|
| 即用性 | 输出可直接进入设计分析阶段，无需二次加工 <sup>[3]</sup> |
| 可追溯性 | 每项分析标注来源（PRD 章节、截图、Figma 页面）<sup>[4]</sup> |
| 准确性 | 基于检查清单，不遗漏关键分析项 <sup>[5]</sup> |
| 规范性 | 固定文档结构，版本管理友好 <sup>[5]</sup> |
| 实战验证 | 质量门禁强制验证，不通过则指明缺失 <sup>[6]</sup> |

## 3. 工作流程设计

### 3.1 流程概览

采用**单 Skill + 显式阶段产物**模式，借鉴 pipeline 设计思想 <sup>[6]</sup>，每阶段生成真实文件：

| 阶段 | 功能 | 输出文件 |
|---|---|---|
| **阶段 1** | 输入解析与结构化 | \`input-structured.md\` |
| **阶段 2** | 需求缺口分析与拆解 | \`gap-analysis.md\` |
| **阶段 3** | 协作对话补充 | \`change-log.md\` |
| **阶段 4** | 质量门禁验证 | \`quality-report.md\` |
| **阶段 5** | 整合输出 | \`final-analysis.md\` |

**选择此模式的理由** <sup>[6]</sup>：
- 每阶段独立输出文档，便于阶段性验证
- 阶段间强制传递，后阶段读取前阶段输出作为输入
- 可在任何阶段停下来检查，提高输出下限
- 统一编号系统，便于追溯

### 3.2 阶段详解

#### 阶段 1：输入解析与结构化

**输入（实际情况）：**

通常情况下，设计师能获取到的输入材料：
- PM 输出的需求清单或 PRD
- 业务背景说明、迭代目标、版本计划

**可选输入（如有）：**
- 界面截图（图片）
- Figma 页面链接
- 用户反馈、数据报表、竞品参考

**输入：**
- PRD 文本（直接粘贴或文件）
- 界面截图（图片）
- Figma 页面链接（通过 Figma MCP 获取）

**处理：**
1. 解析 PRD 文本，提取结构化信息（业务背景、功能点、规则）
2. 分析截图，识别界面元素、交互模式、信息架构
3. 获取 Figma 页面内容，提取设计信息
4. 对每项信息标注来源追溯 <sup>[4]</sup>

**输出：\`input-structured.md\`**

\`\`\`markdown
## Section 1: 业务背景与目标
- [来源: PRD第2节] 业务背景：...
- [来源: PRD第3节] 迭代目标：...

## Section 2: 功能点提取
- [来源: PRD第4节] 功能A：...
- [来源: 截图1] 界面元素：...

## Section 3: 界面信息（来自截图/Figma）
- [来源: 截图1] 页面结构：...
- [来源: Figma页面X] 组件布局：...
\`\`\`

#### 阶段 2：需求缺口分析与拆解

**方法论来源** <sup>[2]</sup><sup>[11]</sup>：

需求分析阶段建议按以下顺序执行：
1. **理解需求原文**：确认 PM 想解决什么问题，以及需求描述是否完整 <sup>[1]</sup>
2. **拆解需求结构**：拆成目标、用户、场景、流程、状态、规则、异常、依赖 <sup>[11]</sup>
3. **澄清歧义和缺口**：把不清楚、冲突、遗漏、不可执行的部分提出来 <sup>[3]</sup>
4. **归纳核心问题**：把零散功能描述提炼成设计可处理的问题定义 <sup>[9]</sup>
5. **形成结构化结论**：输出后续设计分析可直接使用的设计输入 <sup>[10]</sup>

**处理：**
基于检查清单，系统化分析：
1. 用户角色划分
2. 使用场景梳理（主场景、分支场景、异常场景）
3. 边界条件识别
4. 异常状态检测
5. 数据实体提取
6. 信息架构推断
7. 需求缺口标注

**输出：\`gap-analysis.md\`**

\`\`\`markdown
## Section 6: 用户角色分析
| 角色 | 描述 | 来源 |
|---|---|---|
| 普通用户 | [来源: PRD第5节] | ...

## Section 7: 场景清单
### 7.1 主场景
- [✓ 已覆盖] 场景A: ...
- [来源: PRD第6节]

### 7.2 缺失场景
- [✗ 需补充] 场景B: 截图显示XX按钮，但PRD未说明点击后行为

## Section 8: 边界条件
| 边界类型 | 条件值 | 来源 | 状态 |
|---|---|---|---|
| 输入边界 | 最大长度100字符 | [PRD第7节] | ✓ 明确 |
| 数量边界 | 最大100条 | [缺失] | ✗ 需澄清 |
\`\`\`

#### 阶段 3：协作对话补充

**方法论来源** <sup>[12]</sup>：

五段式澄清法通过对话式分析引导需求澄清：
- 问题（Problem）：摒弃用户说的 Y，记录挖掘出的 X
- 目标（Goal）：我们希望达成什么状态
- 价值（Value）：为什么值得做
- 约束（Constraint）：时间、资源、技术、业务约束
- 方案（Solution）：基于以上分析的建议

**处理：**
进入对话模式，引导设计师补充/修正：
1. 呈现缺口清单，逐项询问
2. 记录设计师的补充内容和修正
3. 标注变更原因和追溯

**输出：\`change-log.md\`**

\`\`\`markdown
## 变更记录

### 变更1：补充边界条件
- 时间: 2026-03-31 10:00
- 原因: 阶段2检测到"数量边界"缺失
- 内容: 设计师补充"最大100条，超过提示并禁止添加"
- 来源: 设计师口头说明

### 变更2：修正场景描述
- 时间: 2026-03-31 10:05
- 原因: PRD描述与截图不一致
- 内容: 将场景A修正为...
- 来源: 截图2确认
\`\`\`

#### 阶段 4：质量门禁验证

**方法论来源** <sup>[5]</sup><sup>[10]</sup>：

建议内部评审时重点看以下几个维度：

**正确性**：需求是否真实反映业务和用户问题，是否理解偏差 <sup>[9]</sup>
**完整性**：主流程、异常、边界、依赖、约束是否都被覆盖 <sup>[3]</sup>
**可执行性**：是否能直接支撑后续设计分析和方案设计 <sup>[8]</sup>
**一致性**：同一需求在不同文档、口径、评审中是否一致 <sup>[10]</sup>
**可验证性**：需求是否能通过原型、评审、测试或数据指标验证 <sup>[9]</sup>

**处理：**
基于审核标准验证：

**输出：\`quality-report.md\`**

\`\`\`markdown
## 质量验证报告

### 正确性: ✓ PASS
- 业务问题定义准确

### 完整性: ✗ FAIL
- 缺失项: 空状态处理未定义
- 修正建议: 补充"列表为空时显示引导提示"

### 可执行性: ✓ PASS
- 可直接进入设计分析

### 一致性: ⚠ WARN
- PRD第5节与截图1对按钮命名不一致
- 已在change-log记录修正

### 可验证性: ✓ PASS
\`\`\`

**质量门禁执行规则** <sup>[6]</sup>：
- 全部 PASS → 进入阶段 5
- 有 FAIL → 返回阶段 3 补充
- 有 WARN → 可继续，但需在最终文档标注

#### 阶段 5：整合输出

**方法论来源** <sup>[3]</sup>：

合并前 4 个阶段产物，生成最终需求分析文档，包含 5 项必交付物：
1. 需求分析卡
2. 需求拆解清单
3. 场景与边界说明
4. 待澄清问题清单
5. 需求分析结论

**输出：\`final-analysis.md\`**

## 4. 输出模板设计

### 4.1 需求分析卡模板

**来源** <sup>[3]</sup>

用于快速概览本次需求，建议一页内可读完。

| 字段 | 内容 |
|---|---|
| 需求名称 | |
| 版本/迭代号 | |
| 需求来源 | |
| 背景说明 | |
| 目标用户 | |
| 使用场景 | |
| 核心问题 | |
| 需求拆解 | |
| 业务规则 | |
| 约束条件 | |
| 优先级 | |
| 待澄清项 | |
| 风险点 | |
| 需求结论 | |

### 4.2 需求拆解清单模板

**来源** <sup>[3]</sup><sup>[11]</sup>

把 PRD 或需求清单拆成交互可处理的颗粒度：
- 主任务
- 子任务
- 页面/入口
- 分支流程
- 异常流程
- 状态变化
- 依赖关系

### 4.3 场景与边界说明模板<sup><sup>[3]</sup></sup>

说明"在哪些情况下用户会用、在哪些情况下不能用、有哪些限制"：
- 典型使用场景
- 边界场景
- 不支持场景
- 业务/技术/权限限制

### 4.4 待澄清问题清单模板<sup><sup>[8]</sup></sup>

把未确认内容显式列出，确保需求不带病进入设计：
- 需要 PM 确认的问题
- 需要研发确认的问题
- 需要测试确认的问题
- 需要业务确认的问题

### 4.5 需求分析结论模板<sup><sup>[1]</sup></sup>

用 1 到 3 句话总结本次需求分析结果：
- 谁
- 在什么场景
- 遇到什么问题
- 本次要优先解决什么

## 5. 检查清单设计

### 5.1 需求分析检查清单<sup><sup>[5]</sup><sup>[10]</sup></sup>

\`\`\`markdown
# 需求分析检查清单

## A. 基本信息
- [ ] 需求名称是否明确
- [ ] 版本/迭代号是否标注
- [ ] 需求来源是否记录
- [ ] 业务背景是否说明

## B. 用户与场景
- [ ] 目标用户是否定义
- [ ] 用户角色是否划分
- [ ] 使用场景是否枚举
- [ ] 场景触发条件是否说明

## C. 流程与状态
- [ ] 主流程是否定义
- [ ] 分支流程是否枚举
- [ ] 异常流程是否覆盖
- [ ] 状态变化是否定义

## D. 边界与约束
- [ ] 输入边界是否定义（长度、格式、必填）
- [ ] 数量边界是否定义（最大/最小值）
- [ ] 时间边界是否定义（截止时间、有效期）
- [ ] 权限边界是否定义（角色权限）
- [ ] 数据边界是否定义（数据范围）
- [ ] 技术约束是否标注
- [ ] 业务规则是否明确

## E. 异常状态
- [ ] 错误状态处理是否定义
- [ ] 空状态处理是否定义
- [ ] 加载状态是否考虑
- [ ] 网络异常是否处理
- [ ] 权限不足是否处理

## F. 数据与依赖
- [ ] 数据实体是否识别
- [ ] 数据字段是否定义
- [ ] 依赖关系是否说明
- [ ] 第三方服务依赖是否标注

## G. 信息架构
- [ ] 页面结构是否推断
- [ ] 导航层级是否识别
- [ ] 内容组织是否说明

## H. 可追溯性 <sup>[4]</sup>
- [ ] 每项分析是否标注来源
- [ ] 来源类型是否区分（PRD/截图/Figma/补充）

## 7. Skill 触发与交互设计

### 7.1 触发方式

**Skill 名称**：\`ux-requirements-analyzer\`

**触发条件** <sup>[7]</sup>：
- 用户提及 "需求分析"、"分析 PRD"、"分析需求"、"需求拆解"
- 用户提供 PRD 文档 + 截图/Figma 链接
- 用户请求 "帮我分析这个需求"

### 7.2 用户交互流程

**Step 1：输入接收**

提示用户提供：
- PRD 文本（粘贴或文件路径）<sup>[8]</sup>
- 界面截图（如有）
- Figma 页面链接（如有）

**Step 2：自动分析**

自动执行需求分析 <sup>[2]</sup>，生成中间产物。

**Step 3：协作对话** <sup>[12]</sup>

采用五问法（5 Whys）对话式分析，引导设计师补充缺口。

**Step 4：质量验证**

基于审核标准验证 <sup>[5]</sup><sup>[10]</sup>。

**Step 5：整合输出**

生成最终需求分析文档 <sup>[3]</sup>。

## 9. 成功指标

**来源** <sup>[5]</sup><sup>[10]</sup>

| 指标 | 目标值 | 验证方式 |
|---|---|---|
| 输出完整性 | 检查清单覆盖率 ≥ 90% | 自动校验 |
| 可追溯性 | 100% 项标注来源 | 人工检查 |
| 即用性 | 设计师反馈可直接进入设计分析 | 用户反馈 |
| 准确性 | 缺口识别准确率 ≥ 80% | 与人工分析对比 |
| 规范性 | 文档结构符合模板 | 模板校验 |

## 10. 风险与应对

| 风险 | 影响 | 应对措施 |
|---|---|---|
| Figma MCP 不可用 | 无法获取设计信息 | 提示用户提供截图替代 |
| 截图分析不准确 | 推断错误 | 标注为推断，引导设计师确认 |
| PRD 信息不完整 | 缺口过多 | 强化待澄清清单，引导补充 |
| 设计师不熟悉流程 | 使用困难 | 提供简化引导，逐步推进 |

## 11. 后续迭代方向

1. **可选交付物扩展**：用户路径图、异常流程图、风险清单 <sup>[14]</sup>
2. **历史分析积累**：存储过往需求分析，形成知识库
3. **团队协作**：支持多人协作补充变更
4. **设计分析集成**：与后续设计分析阶段衔接
