# v0.3.3 SKILL.md 重构设计

**日期**: 2026-04-15
**版本**: v0.3.3
**来源**: 用户反馈 + skill-creat-guideline 规范

---

## 背景

当前 v0.3.2b SKILL.md 存在三个问题：

1. **内容庞大** — 873 行，包含大量模板内容嵌入，AI 阅读效率低
2. **模板重复** — SKILL.md 中嵌入完整模板内容，与 `references/需求分析与设计目标说明书模板.md` 重复，违反单一职责原则
3. **流程矛盾** — 工作流概览说「有 P0 缺口 → 停下来补充与探讨」，但 Stage 输出后没有明确的缺口讨论环节描述

---

## 设计目标

1. SKILL.md 精简至 200-300 行，仅保留核心流程指令
2. 模板内容移至 assets/，不再重复嵌入
3. Stage 详细流程拆分为独立 references 文件
4. 统一四段式输出 + 缺口讨论流程
5. 文件命名规范化（全英文，避免中英夹杂）

---

## 设计段 1：目录结构重构

**重构后的 rana 目录结构：**

```
rana/
├── SKILL.md                    (~200-300 行，仅核心流程指令)
├── config.yaml                 (文件解析配置)
├── scripts/
│   └── quality-validator.py    (质量验证脚本)
├── references/
│   ├── stage-1-guideline.md    (Stage 1 详细流程)
│   ├── stage-2-guideline.md    (Stage 2 详细流程)
│   ├── stage-3-guideline.md    (Stage 3 详细流程)
│   ├── p0-gates.md             (所有 Stage P0 缺口规则汇总)
│   ├── analysis-methods.md     (分析方法论：HMW/MVP/五问法/X-Y Problem等)
│   ├── collaboration-protocol.md (协作对话规范、反驳机制)
│   └── _archived-traceability-guide.md (来源标注规范 - 归档，历史遗留)
└── assets/
│   └── analysis-template.md    (需求分析说明书模板)
```

**文件命名规范：**
- 所有文件名使用英文，避免中英夹杂
- 中文内容文件移至 `assets/` 或 `references/`，英文命名，文件开头注明中文标题
- 归档/历史遗留文件使用 `_archived-` 前缀标记

**目录职责说明（来自 skill-creat-guideline）：**

| 目录 | 用途 | 是否加载到 context |
|------|------|---------------------|
| `references/` | 执行时参考的文档 | ✓ 需要加载（agent 读取理解后执行） |
| `assets/` | 输出时使用的文件 | ✗ 不加载（复制/修改后输出） |

## 设计段 3：Stage Guideline 文件结构

**每个 Stage guideline 文件统一结构：**

```markdown
# Stage X Guideline

## 前置条件
（执行本 Stage 的前提）

## 执行步骤
（详细步骤，按顺序）

## 门禁检查
- P0 缺口清单（引用 `references/p0-gates.md`）
- 逻辑合理性诊断（如有）

## 输出：四段式独立消息

**消息块 1（引导语）**：
> 正在为你展示 [阶段描述]

**消息块 2（报告本体）**：
- 读取 final-analysis.md [章节范围]
- 逐字全文输出，不省略、不截断、不使用代码块包裹
- 不输出 [未填充章节]

**消息块 3（缺口清单）**：
- 检查 P0 缺口
- 无缺口：`✓ 无 P0 缺口阻塞`
- 有缺口：呈现缺口表格 + 进入缺口讨论环节

**消息块 4（确认提示语）**：
> [确认语]

## 缺口讨论环节（如触发）

**性质：多轮对话（人在环中）**

缺口讨论不是一次性的问答，而是多轮人机对话过程：
- AI 在分析过程中主动发现问题（逻辑疑点、信息矛盾、P0 缺口）
- AI 主动发起提问，深入挖掘，而非被动等待用户触发
- 每轮对话后判断：缺口是否已补完？是否还有疑点需澄清？

**对话流程**：

```
发现缺口/疑点 → 呈现并提问 → 用户回应 → 
判断是否已补完 → 
  ├─ 已补完：输出确认，进入下一 Stage
  └─ 未补完：继续下一轮提问（循环）
```

**对话规范**：
- 每次只问 1-2 个问题（启用启发式追问，见 `references/collaboration-protocol.md`）
- 使用 HMW 思维提供替代方案（见 `references/analysis-methods.md`）
- 遇用户反驳，执行 5 步反驳机制（见 `references/collaboration-protocol.md`）
- 记录变更到 change-log.md

## 缺口补完确认

- 输出：`✓ 缺口已补完，进入 Stage X+1`
- 不重新输出报告本体
- 更新版本迭代记录

## 输出文件更新
（版本迭代记录、change-log 等）
```

# P0 缺口规则

## Basic 阶段 P0 缺口（阻塞主流程）

|------|------|------|

| 小节 | 字段 | 缺口类型 |
|------|------|---------|
| 3.1 | 现状描述（至少一个核心现象） | P0 |
| 4.1 | 核心用户画像（基础属性 + 痛点） | P0 |
| 4.3 | 核心使用场景（5要素） | P0 |

## Detail 阶段 P0 缺口（阻塞主流程）

| 小节 | 字段 | 缺口类型 |
|------|------|---------|
| 5.1 | 业务北极星指标（指标名称 + 目标值） | P0 |
| 6.3 | 需求全清单（至少列出 P0 需求） | P0 |

## 处理逻辑

- 无 P0 缺口 → 自动进入下一 Stage
- 有 P0 缺口 → 进入缺口讨论环节（多轮对话，人在环中）

1. 模板移至 assets 后，执行时需主动加载（在 guideline 中说明）
2. P0 规则与模板结构需保持同步（模板章节变化时需同步更新 p0-gates.md）
3. 归档文件 `_archived-traceability-guide.md` 保留供内部质量验证参考，不作为 agent 可见参考

5. 缺口讨论性质为「多轮对话（人在环中）」，非一次性问答
