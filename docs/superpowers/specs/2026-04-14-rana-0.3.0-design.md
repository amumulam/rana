# Rana 0.3.0 设计文档 — 输出标准化为字节跳动说明书模板

**版本**: 0.3.0
**日期**: 2026-04-14
**状态**: Draft

---

## 1. 需求概述

### 1.1 核心需求

将 rana 的最终输出从现有的 5 项交付物（A-E）结构，标准化为字节跳动「需求分析与设计目标说明书」9 章节模板，并采用四阶段渐进完善流程（Basic → Core → Detail → Refine）。

### 1.2 关键设计决策

| 项目 | 决策 |
|------|------|
| 输出结构 | 单文件增量式（`需求分析与设计目标说明书.md`） |
| 阶段映射 | Stage 1 → Basic + Core；Stage 3-5 → Detail；Stage 6 挂起 |
| 阶段推进 | 按缺口判断（P0 缺口阻塞，有部分内容可推进） |
| P0/P1/P2 定义 | 按模板章节重新定义 |
| 程序化验证 | 挂起，后续版本实现 |

## 2. 现有结构 vs 新结构对比

### 2.1 现有输出结构（v0.1.0-0.2.0）

```
ux-requirement-analysis/
└── <需求名称>/<YYYY-MM-DD>/
    ├── input-structured.md       ← Stage 1 输出
    ├── gap-analysis.md           ← Stage 2 输出
    ├── change-log.md             ← Stage 3 输出
    ├── quality-report.md         ← Stage 4 输出
    └── final-analysis.md         ← Stage 5 输出（含5项交付物）
```

**5 项交付物**：
- A：需求分析卡（表格卡片）
- B：需求拆解清单（7维度）
- C：场景与边界说明
- D：待澄清问题清单
- E：需求分析结论

### 2.2 新输出结构（v0.3.0）

```
ux-requirement-analysis/
├── _temp/                            ← Stage 0 文件预处理临时输出
│   └── {filename}/auto/{filename}.md
└── <需求名称>/<YYYY-MM-DD>/
    ├── 需求分析与设计目标说明书.md   ← 主输出（单文件增量）
    ├── change-log.md                  ← 协作记录（保留）
    └── quality-report.md              ← AI自评（保留，简化）
```

**9 章节说明书结构**：
- 第一章：前置基础模块（文档基础信息、版本迭代记录）
- 第二章：需求背景（需求来源、历史复盘、影响范围）
- 第三章：核心问题定义（现状描述、根因拆解、问题分级）
- 第四章：用户画像与核心使用场景
- 第五章：业务目标与可量化设计指标
- 第六章：需求清单与优先级排序
- 第七章：核心落地策略（产品/运营/设计）
- 第八章：风险预判与应对方案
- 第九章：各角色重点关注问题

## 3. 四阶段渐进完善流程

### 3.1 阶段定义

模板四阶段要求：

| 阶段 | 核心目的 | 必须完成的小节 |
|------|---------|---------------|
| **Basic** | 搭建文档基础，明确需求背景与前提 | 1.1、1.2、2.1、2.3 |
| **Core** | 精准定位需求核心痛点，明确用户与场景 | 3.1-3.4、4.1-4.5 |
| **Detail** | 明确业务目标、需求优先级、落地策略 | 5.1-5.4、6.1-6.3、7.1-7.4、第八章 |
| **Refine** | 打磨优化 | v0.3.0 不实现 |

### 3.2 与现有 Stage 映射

| 现有 Stage | 四阶段映射 | 说明 |
|-----------|-----------|------|
| Stage 0: 文件预处理 | 独立前置 | 不纳入渐进完善 |
| Stage 1: 输入解析 → **Basic + Core** | 合并为第一阶段输出 |
| Stage 3-5: 协作+整合 → **Detail** | 合并为第三阶段输出 |
| Stage 6: 程序化验证 | 挂起 | 后续版本实现 |

### 3.3 流程图

```
Stage 0: 文件预处理（不变）
        │
        ▼
Stage 1: Basic + Core 输出
        │
        │ 输入材料 → 填充模板第一~四章
        │ 检查 P0 缺口
        │
        ├─ 无 P0 缺口 → 自动进入协作
        └─ 有 P0 缺口 → 停下来补充
        │
        ▼
Stage 2: 协作补充（对应原 Stage 3）
        │
        │ 逐项澄清缺口 → 更新说明书
        │ 版本迭记录表新增条目
        │
        ▼
Stage 3: Detail 输出（对应原 Stage 4-5）
        │
        │ 填充模板第五~八章
        │ 检查 P0 缺口
        │ AI 自评 → quality-report.md
        │ 向用户呈现完整说明书
        │
        ▼
Stage 4: 程序化验证（挂起）
```

## 4. P0/P1/P2 缺口定义

### 4.1 分级原则

| 优先级 | 定义 | 影响 |
|-------|------|------|
| **P0** | 缺失会导致文档无法交付设计阶段 | 阻塞进入下一阶段 |
| **P1** | 缺失会影响设计决策质量 | 可带病推进，协作阶段补充 |
| **P2** | 缺失不影响核心交付 | 后续迭代补充，不阻塞 |

### 4.2 各阶段 P0 小节

**Basic 阶段 P0：**
- 1.1 文档基础信息：需求名称、所属业务线/产品、迭代版本号
- 2.1 需求来源：至少一项溯源依据（数据/反馈/战略）
- 2.3 影响范围：覆盖用户群体

**Core 阶段 P0：**
- 3.1 现状描述：至少一个核心现象
- 4.1 核心用户画像：基础属性 + 痛点与诉求
- 4.3 核心使用场景：用户 + 场景 + 任务 + 障碍 四要素

**Detail 阶段 P0：**
- 5.1 业务北极星指标：指标名称 + 目标值
- 6.2 需求全清单：至少列出 P0 需求

### 4.3 P1/P2 小节

**P1 小节（重要但可带病推进）：**

| 章节 | P1 小节 |
|------|---------|
| 第一章 | 1.2 版本迭代记录 |
| 第二章 | 2.2 历史复盘 |
| 第三章 | 3.2 根因拆解、3.3 问题分级 |
| 第四章 | 4.2 次要用户画像、4.4 次要/边缘场景 |
| 第五章 | 5.2 可量化设计过程指标、5.3 体验兜底指标 |
| 第六章 | 6.1 需求排序模型 |
| 第七章 | 7.2-7.4 产品/运营/设计策略 |
| 第八章 | 风险预判表格 |

**P2 小节（后续补充）：**
- 第九章：各角色重点关注问题

## 5. Stage 1 逻辑调整

### 5.1 输入材料 → Basic + Core 输出

Stage 1 输入材料后，根据材料充分度：
1. 自动填充可完成的 Basic + Core 小节
2. 无法完成的标记 `[缺失]`
3. 输出到 `需求分析与设计目标说明书.md`
4. 检查 P0 缺口：
   - 无 P0 缺口 → 自动进入 Stage 2（协作补充）
   - 有 P0 缺口 → 告知用户，等待补充材料或协作澄清

### 5.2 材料充分度判断

| 材料类型 | 可填充的小节 |
|---------|-------------|
| PRD 文本 | 2.1 需求来源、2.3 影响范围、3.1 现状描述 |
| 界面截图 | 3.1 现状描述（补充视觉现象） |
| 业务背景说明 | 1.1 文档基础信息、2.3 影响范围 |
| 用户数据/反馈 | 4.1 核心用户画像 |
| 口头说明 | 作为 `[用户确认]` 来源引用 |

### 5.3 来源标注沿用

现有来源标注格式继续使用：
- `[PRD第X节]` / `[截图X]` / `[用户确认]` / `[缺失]`
- 模板表格中的"来源追溯"列填写来源标注

## 6. Stage 2（协作补充）逻辑调整

### 6.1 协作目标变化

**现有**：补充 33 项检查清单的缺口
**新增**：补充说明书模板各章节的 `[缺失]` 项

### 6.2 协作流程

1. 呈现 P0 缺口清单（按阶段分组）
2. 逐项澄清（延续现有 Stage 3 对话脚本）
3. 更新说明书对应章节
4. 版本迭代记录表新增条目：
   ```
   | V1.1 | YYYY-MM-DD | AI | 补充核心用户画像、核心使用场景 | 第三章、第四章 |
   ```
5. 检查 P0 缺口是否全部补完 → 进入 Stage 3

### 6.3 change-log.md 保留

继续记录协作澄清过程，格式不变。

---

## 7. Stage 3（Detail 输出）逻辑调整

### 7.1 前置条件

- Basic + Core 阶段 P0 缺口已全部补完
- 说明书第一~四章已填充

### 7.2 Detail 输出流程

1. 基于已有信息 + 协作补充内容
2. 填充第五~八章（业务目标、需求清单、策略、风险）
3. 无法完成的标记 `[缺失]` 或 `[待确认]`
4. 检查 Detail 阶段 P0 缺口
5. AI 自评 → 输出 `quality-report.md`
6. 向用户呈现完整说明书

### 7.3 quality-report.md 简化

**现有格式**：5维度自评 + P0 缺口状态 + 程序化验证结果
**新增格式**：

```markdown
# 质量门禁报告

**检查日期**：YYYY-MM-DD
**当前版本**：V1.X

## 四阶段完成状态

| 阶段 | 状态 | P0 缺口 | 说明 |
|------|------|---------|------|
| Basic | ✓ 完成 | 0 | 第一~二章已填充 |
| Core | ✓ 完成 | 0 | 第三~四章已填充 |
| Detail | ⚠ 部分 | 2 | 第五、六章有 P1 缺口 |

## P0 缺口汇总

（若无 P0 缺口则写"无 P0 缺口阻塞"）

## 结论

说明书已输出，可交付设计分析阶段。
Detail 阶段 P1 缺口建议在后续评审中补充。
```

## 8. 模板章节与角色分工

### 8.1 填写责任

| 章节 | 主要填写方 | AI 辅助程度 |
|------|-----------|------------|
| 第一~二章 | AI 自动提取 | 高 |
| 第三章 | AI 分析 + 用户确认 | 中 |
| 第四章 | AI 推断 + 用研数据补充 | 中 |
| 第五~六章 | AI 分析 + PM 确认 | 中 |
| 第七章 | AI 生成框架 + 各角色补充 | 低 |
| 第八章 | AI 预判 + 各角色补充 | 低 |
| 第九章 | 各角色填写 | 无（AI 不生成） |

### 8.2 模板引用

说明书结构参照 `ref/需求分析与设计目标说明书模板.md`，每章节填写说明必须遵守模板要求。
