# Validator Round 2 — 设计文档

**日期**：2026-03-31
**目标**：解决 quality-validator.py 的两个遗留问题：高误报模糊词 + 缺失字段数量检查

---

## 背景

上一轮优化（7项变更）完成后，validator 在 test-b-edge 场景产生 2 处误报（WARN），且从未检查需求分析卡的字段完整性。

### 问题 A：模糊词高误报

test-b-edge 触发的 2 处 WARN 均为误报：

| 文件 | 行 | 词 | 误报原因 |
|------|----|----|---------|
| `final-analysis.md:75` | `通常` | 位于 `[推断：...通常用于...]` 来源标注内，属于推断依据，不是需求文本 |
| `input-structured.md:39` | `相关` | 是复合词 `相关度算法` 的一部分，是精确技术术语 |

根本原因：`相关/一般/通常/正常情况下` 这 4 个词在真实输出中误报率高，而它们的漏报成本低（真正模糊的需求通常同时触发完整性维度的 FAIL）。

### 问题 B：缺失字段数量检查

`templates/requirement-card.md` 定义 14 个必填字段，但 validator 完全不检查字段是否存在。

test-b-edge 实际输出了 19 个字段（多 6 个自定义字段，少 `需求拆解` 1 个），validator 未检测到任何问题。

## 详细设计

### 变更 1a：删除高误报模糊词

从 `VAGUE_TERMS` 列表中删除以下 4 项：

```python
"相关",
"一般",
"通常",
"正常情况下",
```

**保留的 12 项**（均为高置信度的模糊需求指示词）：

```python
VAGUE_TERMS = [
    "适当",
    "合理",
    "友好",
    "简洁",
    "清晰",
    "尽量",
    "尽可能",
    "TBD",
    "待定",
    "暂定",
    "后续确认",
    "后续讨论",
]
```

在 VAGUE_TERMS 定义上方添加注释，说明删除理由：

```python
# 模糊表述（不可执行词汇）
# 注：以下词汇已因高误报率移除：
#   "相关"      — 常出现在「相关度算法」等技术术语中
#   "一般"      — 叙述性语言中误报率高
#   "通常"      — 常出现在来源标注的推断依据中
#   "正常情况下" — 分析注释中的背景描述会被误拦
# 这些词的漏报成本低：真正模糊的需求通常同时触发完整性维度 FAIL
```

### 变更 1b：新增字段完整性检查

#### 新增常量

```python
# 需求分析卡必填字段（来自 templates/requirement-card.md）
REQUIRED_CARD_FIELDS = [
    "需求名称", "版本/迭代号", "需求来源", "背景说明",
    "目标用户", "使用场景", "核心问题",
    "需求拆解", "业务规则", "约束条件", "优先级",
    "待澄清项", "风险点", "需求结论",
]
```

#### 新增函数 `check_card_fields(content)`

**逻辑**：

1. 扫描 `final-analysis.md` 内容，定位 `交付物 A` 或 `需求分析卡` 章节（`##` 级别）
2. 提取该章节内所有表格数据行的第一列（字段名）
3. 剥除 `**` 粗体标记（test-a 风格为 `| **需求名称** |`，test-b 风格为 `| 需求名称 |`）
4. 跳过表头行（首列匹配 `字段` 等关键词）和分隔行
5. 遇到下一个 `##` 标题即停止（章节结束）

**判定规则**：

| 情况 | 状态 | 是否阻断 |
|------|------|---------|
| 所有 14 个必填字段全部存在，无多余 | PASS | 否 |
| 所有 14 个必填字段存在，但有多余字段 | WARN | 否（允许 AI 扩展） |
| 缺少任意必填字段（无论是否有多余） | FAIL | 是 |
| 找不到需求分析卡章节 | FAIL | 是 |

**返回格式**：

```python
{
    "missing": ["需求拆解"],          # 必填但缺失的字段
    "extra": ["功能入口", "主流程"],   # 存在但不在必填列表的字段
    "found": ["需求名称", ...],        # 实际找到的所有字段
    "status": "PASS" | "WARN" | "FAIL"
}
```

#### 输出格式（接入 run_validation）

在 `final-analysis.md` 的章节完整性检查之后输出：

```
  字段完整性: ✓ PASS（14/14）
  字段完整性: ⚠ WARN（14/14 必填字段存在，另有 5 个扩展字段：功能入口, 主流程...）
  字段完整性: ✗ FAIL（缺失必填字段：需求拆解, 待澄清项）
```

FAIL 时加入 `failed_dimensions`，影响最终 PASS/FAIL 判定。

## 预期测试结果

| 场景 | 变更前 | 变更后 |
|------|--------|--------|
| test-a-normal | PASS，0 WARN | PASS，0 WARN，字段完整性 PASS（14/14） |
| test-b-edge | PASS，2 WARN（误报） | PASS，1 WARN（字段完整性：19字段，5个扩展） |

test-b-edge 的模糊词 2 处 WARN 消除；字段完整性产生 1 处新 WARN（19 个字段，6 个多余，1 个缺失 `需求拆解`）——等等，缺失 `需求拆解` 应为 FAIL。

**修正**：test-b-edge 字段实际情况：

- 缺失：`需求拆解`（1个）→ FAIL
- 多余：`所属产品/系统`, `搜索对象`, `功能入口`, `主流程`, `技术平台`, `迭代目标`（6个）→ WARN

所以 test-b-edge 字段完整性为 **FAIL**（缺失 `需求拆解`）。这是一个真实问题——test-b-edge 的分析卡确实遗漏了 `需求拆解` 字段。整体 validator 结果从 PASS 变为 FAIL，这是正确行为（test-b-edge 本身就是一个边缘场景测试）。

| 场景 | 变更前整体 | 变更后整体 |
|------|-----------|-----------|
| test-a-normal | ✓ PASS | ✓ PASS |
| test-b-edge | ✓ PASS（含 2 WARN） | ✗ FAIL（字段缺失：需求拆解） |

test-b-edge 结果从 PASS 变 FAIL 是**预期的正确行为**——它检出了之前漏过的真实问题。
