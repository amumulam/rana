## 背景

- **A 类（一致性）**：文档间互相矛盾，不影响已有输出，但影响长期维护

本次优化同时解决两类问题。

## 详细设计

### 变更 1：analysis-checklist.md — 优先级标注

**目的**：让 Stage 4「P0 缺口是否已补完」的判断有明确文档依据，消除 AI 隐式推断。

**优先级定义**（加在文件开头）：

```
优先级含义：
(P0) 缺失则无法判断功能方向或用户是谁，设计无法开始
(P1) 缺失影响设计完整性，进入详细设计前必须确认
(P2) 缺失不阻断设计，开发前确认即可
```

**各条目标注**：

| 类别 | 条目 | 优先级 |
|------|------|--------|
| A. 基本信息 | 需求名称是否明确 | P1 |
| | 版本/迭代号是否标注 | P2 |
| | 需求来源是否记录 | P1 |
| | 业务背景是否说明 | P0 |
| B. 用户与场景 | 目标用户是否定义 | P0 |
| | 用户角色是否划分 | P1 |
| | 使用场景是否枚举 | P0 |
| | 场景触发条件是否说明 | P1 |
| C. 流程与状态 | 主流程是否定义 | P0 |
| | 分支流程是否枚举 | P1 |
| | 异常流程是否覆盖 | P1 |
| | 状态变化是否定义 | P1 |
| D. 边界与约束 | 输入边界是否定义 | P1 |
| | 数量边界是否定义 | P1 |
| | 时间边界是否定义 | P1 |
| | 权限边界是否定义 | P0 |
| | 数据边界是否定义 | P1 |
| | 技术约束是否标注 | P2 |
| | 业务规则是否明确 | P1 |
| E. 异常状态 | 错误状态处理是否定义 | P0 |
| | 空状态处理是否定义 | P0 |
| | 加载状态是否考虑 | P1 |
| | 网络异常是否处理 | P1 |
| | 权限不足是否处理 | P1 |
| F. 数据与依赖 | 数据实体是否识别 | P1 |
| | 数据字段是否定义 | P2 |
| | 依赖关系是否说明 | P1 |
| | 第三方服务依赖是否标注 | P2 |
| G. 信息架构 | 页面结构是否推断 | P1 |
| | 导航层级是否识别 | P1 |
| | 内容组织是否说明 | P2 |
| H. 可追溯性 | 每项分析是否标注来源 | P0 |
| | 来源类型是否区分 | P1 |

P0 共 8 项：业务背景、目标用户、使用场景、主流程、权限边界、错误状态处理、空状态处理、每项标注来源。

### 变更 2：traceability-guide.md — 推断标注规范统一

**目的**：消除 `[推断]` 裸标签与 `[来源推断]` 之间的歧义；正式定义分析方法推断格式。

**变更内容**：

1. 来源类型表中，「设计师推断」行更新为：

   | 设计师推断 | `[来源推断]` | `[截图1推断：截图显示移动端界面]` | 必须带推断依据，裸 `[推断]` 不合法 |

2. 新增「分析方法推断」子类型：

   | 分析方法推断 | `[方法名推断：依据]` | `[场景还原推断：基于PRD第1节用户描述]` | 五问法/X-Y分析/场景还原得出的推断 |

3. 在「标注原则 — 推断必须带依据」小节补充示例：

   ```markdown
   **分析方法推断示例**：
   - [五问法推断：追问至第3层，真实需求是快速定位未读消息]
   - [X-Y分析推断：PM 提案是消息中心（Y），真实问题是消息感知缺失（X）]
   - [场景还原推断：基于PRD第1节「用户反映错过消息」的心境还原]
   ```

4. 在「快速参考」末尾补充分析方法推断条目：
   ```
   [场景还原推断：依据]  ← 场景还原分析得出
   [五问法推断：依据]    ← 五问法分析得出
   [X-Y分析推断：依据]  ← X-Y Problem 分析得出
   ```

### 变更 3：SKILL.md — 多处更新


五个交付物全部按此模式处理。

#### 3b. 来源标注汇总表对齐

第476行汇总表「设计师推断」行由 `[推断]` 改为 `[来源推断]`，新增「分析方法推断」行：

| 分析方法推断 | `[方法名推断：依据]` | `[场景还原推断：基于PRD第1节]` |

#### 3c. Stage 3 — 加冲突讨论步骤

在 Step 3.3 后插入冲突处理分支：

```
Step 3.3 补充——若确认结果与已有信息冲突：

> [确认方A] 说「X」，[确认方B] 说「Y」，两方信息存在冲突。
> 建议 [确认方A] 和 [确认方B] 对齐讨论后确认最终值。
> 我先在 change-log 中记录冲突状态，等讨论完成后更新。

在 change-log 中记录冲突格式：

### CHG-N：[描述]（冲突待讨论）
- **冲突**：[确认方A] 说「X」`[确认方A确认]`；[确认方B] 说「Y」`[确认方B确认]`
- **状态**：⚠ 待讨论，需 [确认方A] + [确认方B] 对齐后确认
- **内容**：[讨论完成后填写]
- **影响**：[讨论完成后填写]

讨论完成后将内容补全，来源标注为：
`[原来源A，已与原来源B冲突，以确认方C为准]`
```

#### 3d. Stage 4 — 加「对话穷尽」分支

执行规则新增第三条分支：

```
- 有 FAIL，且 Stage 3 协作对话已穷尽（确认方明确表示无法提供更多信息）
  → 可继续生成 Stage 5，但整体质量状态保持 FAIL
  → 交付物 D（待澄清清单）为本次核心产出
  → 在 quality-report.md 和 final-analysis.md 开头明确标注「当前质量状态：FAIL，信息不足，待澄清后重新分析」
```

#### 3e. 多功能 PRD 说明

在工作流概览后新增一节「多功能需求处理」：

```markdown
## 多功能需求处理

当 PRD 包含多个独立功能点时，按以下方式组织：

**Stage 1**：提取时识别功能点清单，向用户确认功能边界和数量。

**Stage 2**：gap-analysis.md 按功能点分节，每节独立执行7维度拆解和缺口分析：
  ## 功能一：[名称]
  ## 功能二：[名称]
  检查清单在文件末尾统一汇总（合并所有功能点通过率）。

**Stage 4**：质量门禁基于合并通过率；P0 缺口按功能点独立判断（每个功能点的 P0 缺口均需补完）。

**Stage 5**：
  - 交付物 A（需求分析卡）× 每功能点一份
  - 交付物 B（拆解清单）× 每功能点一份
  - 交付物 C（场景边界）× 每功能点一份
  - 交付物 D（待澄清清单）× 1份（跨功能点汇总）
  - 交付物 E（分析结论）× 1份（整体结论）
```
